# PY7-S3-framework-binding-and-bar-b-gate — Framework-binding exercise + the two-gate bar-B proof

## Why

Closes the epic: exercises the framework binding (PY6 — a request-scoped distinct_id carried through the middleware) and stands up **the bar-B proof itself** — the two gates the architect ruled are together the Python analog of TS Fernly's typecheck-against-dist: a **fidelity gate** (mypy against the installed distribution's public types) + an **enforcement gate** (an AST import-audit asserting the example imports ONLY the public API, no internals). It is the Python realization of TS `E10-S8` (React wiring) + the bar-B/bar-A proof that closes E10.

## Scope

### In

- **Framework-binding exercise:** drive the PY6 middleware in-process against the SAME recording adapter from PY7-S1 — send a request, the middleware opens a `new_context()`, the handler binds `set_context_distinct_id(...)` + `add_tag(...)` and `context(analytics).capture(...)`s through the context-scoped view, and the assertion is that the recorded `NeutralEvent` carried the request-bound `distinct_id` + tags. **Match the PY6 precedent EXACTLY (read `tests/test_django_middleware.py` + `tests/test_asgi_middleware.py`) — do NOT invent a full Django settings/`urls.py`/`Client` harness:**
  - **Django (WSGI):** the `RequestContextMiddleware` is a plain callable — drive it directly (`RequestContextMiddleware(view)(request)`) with a bare request stand-in, NO Django `settings.py`/`urls.py`/`django.test.Client` needed. `django` is the dev-dep only so `_DJANGO_AVAILABLE` is `True` and the middleware constructs.
  - **ASGI (FastAPI/Starlette):** drive a real Starlette app over `httpx.ASGITransport` inside `anyio.run(...)` (Starlette's own `TestClient` pairing is deprecated — the PY6 test uses `httpx.ASGITransport` + `httpx.AsyncClient`). `starlette` + `httpx` are the dev-deps.
  - Import the middleware from the PUBLIC point: `from analytics_kit.integrations import RequestContextMiddleware` / `RequestContextASGIMiddleware` (lazily re-exported via `__getattr__`), NOT from the deeper `analytics_kit.integrations.django`/`.asgi` submodules — the deeper import would FAIL the S3 AST audit (see Enforcement gate). Do the same for `context`/`set_context_distinct_id`/`add_tag` (all in `analytics_kit.integrations.__all__`).
  No bound socket (both paths are in-process request simulators). Pick ONE framework to exercise (Django OR ASGI) — the epic requires "a framework binding," not both; the other is available as an additive expansion.
- **The bar-B fidelity gate:** `uv run mypy` in the example project type-checks the whole example (taxonomy + config + recording adapter + the capture/query/framework exercise) against the INSTALLED `analytics-kit` public types (via the editable dep + `py.typed`), with ZERO `analytics_kit` edits. A clean run IS the bar-B proof of config-only adoption.
- **The bar-B enforcement gate:** an **AST import-audit test** (in the example project) that `ast.parse`s every example `*.py`, walks `ast.Import` + `ast.ImportFrom` nodes (handling `from x import y as z` aliases and multi-line imports — that's WHY it's AST, not grep), and asserts every `analytics_kit` import resolves to a PUBLIC namespace only:
  - **allowed:** `import analytics_kit` / `from analytics_kit import <name>` (the top-level namespace — its `__all__` is the public surface, verified to expose everything the example needs: `create_analytics`, `AnalyticsAdapter`, `ConsentState`, `NeutralEvent`, `NeutralResponse`, the taxonomy surface `define_taxonomy`/`enforce_allowlist`/`ViolationPolicy`, the full query surface, etc.), AND the three curated public SUBPACKAGES `analytics_kit.integrations` / `analytics_kit.query` / `analytics_kit.server` (each verified to have its own `__all__`) — i.e. `from analytics_kit.integrations import RequestContextMiddleware` is allowed (the middleware's honest public import point, lazily re-exported via `integrations.__getattr__`);
  - **FAIL:** any import DEEPER than those (`analytics_kit.provider`, `analytics_kit.factory`, `analytics_kit.integrations.django`, `analytics_kit.integrations.asgi`, `analytics_kit.integrations.context`, `analytics_kit.query.http_adapter`, `analytics_kit.query.client`, `analytics_kit.server.adapter`, `analytics_kit.server.wire_mapper`), any `_WIRE_*` name, any `_`-prefixed name imported from a public namespace. Note: the middleware submodules `.django`/`.asgi` and the context accessors' home `.context` are DEEPER than `analytics_kit.integrations` and MUST fail — the example imports the middleware + accessors from `analytics_kit.integrations` (top-level), never the submodule (this is why the S3 framework exercise pins the public import point).
  AST, not grep (multi-line/aliased imports; the codebase already pins structural guarantees with real parsing — see the `inspect.getsource` fences in the PY6 tests).

## Technical notes

- **#4 ARCHITECT RULING (2026-07-10) — the TWO-GATE model (the load-bearing mechanic):** the TS bar-B gate is enforced by PHYSICAL ABSENCE (`dist` only has re-exports, so `tsc` can't resolve internals). Python has NO such boundary — an editable/wheel install exposes the whole import tree, and `py.typed`/`__all__`/mypy do NOT block a deep import. So the Python bar-B gate SPLITS into two, and neither alone suffices:
  1. **Fidelity gate** — mypy in the separate example project resolving the installed `analytics-kit` public types (via `py.typed`), zero library edits → proves "type-checks against the installed distribution."
  2. **Enforcement gate** — the AST import-audit → proves "public API only, no internals reached." mypy/`__all__`/packaging each PROVABLY cannot enforce this alone (mypy resolves `import analytics_kit.provider` happily; the internals can't be excluded from the wheel because the public factories need them at runtime; there's no Python `exports`-map). Say this explicitly so nobody later "simplifies" it back to a single gate and silently deletes enforcement.
- **The AST audit MUST allowlist the curated public subpackages** (verified against the real code): the exact allowed module-path set is `{"analytics_kit", "analytics_kit.integrations", "analytics_kit.query", "analytics_kit.server"}` — the top-level namespace plus the three subpackages that each expose an `__all__` (`integrations/__init__.py`, `query/__init__.py`, `server/__init__.py`). `analytics_kit.integrations` is the middleware's honest public import point (the middleware classes are lazily re-exported from it via `__getattr__` — an import FROM `analytics_kit.integrations` is public; an import FROM `analytics_kit.integrations.django` is a deeper violation). Encode this set as a literal in the audit. Rule: an `analytics_kit`-rooted import is ALLOWED iff its dotted module path is exactly one of the four; FORBIDDEN if it's deeper (a longer dotted path) OR if any imported NAME is `_`-prefixed / matches `_WIRE_*`. Without the subpackage allowlist the audit false-positives on the legitimate `from analytics_kit.integrations import RequestContextMiddleware` the framework exercise needs.
- **Negative control is a PARSED STRING, not a live import.** The control asserts the audit function FLAGS a deliberately-internal import; author it as a source string handed to the audit's parse routine (e.g. `_INTERNAL_SAMPLE = "from analytics_kit.provider import Analytics"`), NOT as a real import statement in a shipped example module — a real internal import would itself fail the fidelity gate AND break the "diff touches only public-consuming code" bar-B invariant. This also proves the audit's parse path handles the `from ... import ...` form (the aliased/multi-line handling is the reason it's AST not grep).
- **Framework injection (architect (c)) — match the PY6 precedent, don't invent a heavier harness.** Read `tests/test_django_middleware.py` + `tests/test_asgi_middleware.py`: the Django path drives the `RequestContextMiddleware` as a PLAIN CALLABLE with a bare request stand-in (NO `django.test.Client`, NO `settings.py`/`urls.py`); the ASGI path drives a real Starlette app over `httpx.ASGITransport` + `httpx.AsyncClient` inside `anyio.run(...)` (Starlette's own `TestClient` is deprecated). In both, the handler binds `set_context_distinct_id(...)`/`add_tag(...)` inside the middleware-opened context and `context(analytics).capture(...)` records on the PY7-S1 recording adapter in-memory, no network. This re-drives PY6's own middleware tests at the CONSUMER layer — copy the PY6 harness shape, don't reinvent it.
- **Fidelity-gate run-invocation (pin, from S1):** the fidelity gate runs as `cd python/examples/<product> && uv run mypy .` — a SEPARATE invocation from the main `python/` gates (whose `[tool.mypy] files = ["src","tests"]` deliberately exclude `examples/**`). The enforcement gate (AST audit) + the exercise run under `uv run pytest` in the same example dir. Both belong in the documented bar-B proof note.
- **No seam defect (architect confirmed):** every capability the example needs is reachable from the public `analytics_kit` namespace (`create_analytics`, `AnalyticsAdapter`, `NeutralEvent`, the query surface incl. `QueryTransport`/`QueryClientConfig`/`create_query_client`, the taxonomy surface, the server factory) + the public `analytics_kit.integrations` middleware. The example implements `AnalyticsAdapter`/`QueryTransport` purely from re-exported public types — zero `analytics_kit` edit. If a future finding contradicts this, it's a seam bug to surface, not an example patch.
- **CONTRACT reference (port TO):** `ts/examples/fernly/src/{bar-b-config-only.test.ts, bar-a-adapter-swap.test.ts, app/*}` — the bar-B on-disk structural assertion (Fernly's no-`paths`-reroute check is the analog of the "no `../../src` reroute" + the import-audit), the framework wiring, config-only adoption. Server-shaped (no browser/JSX component tree — the middleware is the server analog of the React provider).
- **Neutrality:** `examples/**` exempt from the scan; the two gates enforce public-API-only + config-only.
