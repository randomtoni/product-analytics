# PY6-S1-context-core-and-scoped-view ‚Äî `contextvars` request scope, `@scoped`, and the context-aware capture path

## Why

The `contextvars` request scope is the server analog of the React provider/hooks: it carries a `distinct_id` + super-prop-like tags per request/task, so a request handler can `capture("event", {...})` without threading the distinct id through every call. This story builds the framework-agnostic context core (the middleware in S2/S3 just drives it) + the context-aware capture entry that reads it ‚Äî WITHOUT mutating the shipped seam provider (the architect ruling). It is the Python realization of TS `E9-S2` (the create-once client + context-scoped identity pattern), server-shaped, de-branded from `posthog-python/posthog/contexts.py`.

## Scope

### In

- `analytics_kit/integrations/context.py` (fills the empty PY1 `integrations/` package) ‚Äî the framework-agnostic `contextvars` stack, de-branded from `posthog-python/contexts.py` (keep ONLY `distinct_id` + `tags`; drop the exception-autocapture / session-id / device-id / code-variables machinery ‚Äî PostHog error-tracking + browser concepts with no server-neutral home):
  - A module-private `ContextVar` holding the active context scope (a small `ContextScope` object carrying `distinct_id: str | None` + `tags: dict[str, object]`).
  - `new_context(fresh: bool = False)` ‚Äî a `@contextmanager` that opens a child scope (inheriting the parent's `distinct_id` + tags) or, with `fresh=True`, an isolated scope; yields it, and restores the previous scope on exit (the posthog-python `new_context` pattern; `contextvars` token reset ‚Äî see AC for nesting/`fresh` semantics). Drop posthog-python's `capture_exceptions`/`client` params (error-tracking machinery, out of scope).
  - Accessors: set/read the active `distinct_id`, `add_tag(key, value)` to add a request-scoped tag, and a current-context reader.
  - `@scoped(fresh: bool = False)` ‚Äî a decorator that wraps a function in a `new_context()` (the posthog-python `@scoped` pattern) so a task/handler runs inside a context. **Async-aware** (dual sync/async wrapper ‚Äî see AC): decorating an `async def` returns an `async def` wrapper that `await`s the coroutine inside the live scope. Drop the posthog-python `client`/`capture_exceptions` args.
- **The context-aware capture path** (the #4 architect ruling ‚Äî Option B + context read in the binding layer, NOT the seam provider):
  - **Consumer-facing shape (PM-locked 2026-07-10 ‚Äî Shape A, NOT a hedge):** a FREE accessor `context(analytics: Analytics) -> ScopedContextView` in `integrations/context.py` returning a small view bound to the passed shipped `Analytics` instance. The view exposes `capture(event, properties=None, *, distinct_id=None, dedupe_id=None)` ‚Äî the shipped `provider.capture` shape MINUS the required `distinct_id` (the context supplies it). Consumer call: `scoped = context(analytics); scoped.capture("event", {...})` ‚Äî the handle is captured ONCE, no distinct_id and no `analytics` re-threaded per call (that's the ergonomic the epic exists for). **NOT** `context(name)` (no profile-name arg ‚Äî the TS name‚Üíenrichment-profile lookup is N-A-by-platform, epic Out-of-scope); **NOT** `.track`/`.page` (TS-browser verbs ‚Äî server target has `capture` only); **NOT** an accessor ON `Analytics` (that touches the frozen provider surface the #4 ruling locked ‚Äî it MUST be a free function taking the instance as an arg); **NOT** a module-level `capture_in_context(analytics, ...)` function (rejected ‚Äî it re-threads the handle every call). The verb name `context` collides with TS `context()` intentionally (same request-scoped-view ROLE); the argument differs by platform (instance vs profile-name) and that divergence is correct (PY2 node baseline).
  - The view reads the active context and calls the SHIPPED provider's unchanged `capture(resolved_distinct_id, event, merged)` where `merged = {**context_tags, **(properties or {})}`:
    - resolves `distinct_id` = the explicit `distinct_id=` keyword arg on `view.capture` if given (the escape hatch ‚Äî a handler capturing for a DIFFERENT user than the request's ambient identity), else the active context's `distinct_id`; **raises a clear neutral error** ("no active analytics context and no explicit distinct_id") when NEITHER is present (the binding raises ‚Äî the base provider's required-arg contract is untouched).
    - merges ONLY the context tags into the call-time `properties` (tags first, call-time props win) and hands the result to `provider.capture`. **The binding does NOT touch `super_properties`** ‚Äî it can't (they're private to the provider). The provider's own `_merge_super_properties` then prepends the super-props (`{**super, **{**tags, **call}}`), so the FINAL bag `provider.capture` gates is `{**super_properties, **tags, **call_properties}` ‚Äî the full three-tier precedence is a composition of the binding's tag-merge and the provider's existing super-merge, NOT something the binding builds alone. See Technical notes (b).
  - **The shipped `provider.py` is NOT modified** ‚Äî it stays context-agnostic; the context read lives entirely in this binding layer. Because the merged bag flows THROUGH the unmutated `provider.capture`, the tags automatically cross `provider._allowed` (the allowlist gate) and `validate_event_props` (the taxonomy validator) ‚Äî that is precisely how "tags are gated" holds without touching the provider.

### Out

- The Django middleware ‚Äî **PY6-S2** (drives `new_context()` per request).
- The ASGI/FastAPI middleware ‚Äî **PY6-S3**.
- Any modification to the shipped `provider.py` / the seam ‚Äî explicitly NOT done (the architect ruling; the base provider stays context-agnostic).
- Library-computed context fields (route, request-id, etc.) ‚Äî **out of PY6 scope** (PM-confirmed): PY6 carries CONSUMER TAGS ONLY, all gated. Auto-attaching library-computed request metadata is a deliberate additive follow-up (it introduces a computed-not-supplied lane that attaches after the gate).
- Flask / Celery bindings ‚Äî deferred (PM-locked, additive-by-config).

## Acceptance criteria

- [ ] `new_context()` opens a context scope carrying `distinct_id` + `tags`, yields it, and restores the previous scope on exit (`contextvars` token reset ‚Äî no leak across requests). **Nesting semantics (port `contexts.py:69-90`):** a nested `new_context()` INHERITS the parent's `distinct_id` + tags by default; a child that sets its own `distinct_id` overrides for its scope; child tags take precedence over parent tags on collision (parent-first collect, then child-update). A `fresh=True` flag (default `False`) starts an isolated scope that does NOT inherit the parent's `distinct_id`/tags (port the `fresh` param ‚Äî needed so a decorated task can opt out of an ambient request identity). Exiting a child restores the parent's `distinct_id` + tags exactly.
- [ ] `@scoped` runs a decorated function inside a `new_context()`, and is **async-aware**: applied to an `async def`, it returns an `async def` wrapper that opens the (synchronous) `new_context()` and `await`s the coroutine so the scope is live for the coroutine's ACTUAL execution (NOT torn down before it runs ‚Äî the footgun a sync-only wrapper on an async handler hits); applied to a sync function, it returns a sync wrapper. Both preserve the wrapped function's metadata (`functools.wraps`). De-branded from `posthog-python/contexts.py:452-479` (drop its PostHog-specific `client`/`capture_exceptions` args per "keep distinct_id+tags, drop the rest").
- [ ] `add_tag(k, v)` adds a request-scoped tag readable within the active context.
- [ ] A free `context(analytics: Analytics) -> ScopedContextView` accessor lives in `integrations/context.py`; the view exposes `capture(event, properties=None, *, distinct_id=None, dedupe_id=None)` and delegates to the passed instance's UNMUTATED `provider.capture`. `context()` is NOT a method on `Analytics` (the frozen surface is untouched); a test asserts `Analytics`'s public surface gained no `context`/`capture_in_context` member.
- [ ] The view's `capture` resolves `distinct_id` = explicit `distinct_id=` kwarg else active-context `distinct_id`; with NEITHER present it raises a clear neutral error (from the binding ‚Äî NOT a silent no-distinct-id capture, NOT a personless-UUID fallback). Explicit kwarg overrides the context id (the escape hatch).
- [ ] Context tags merge into `properties` with final precedence `super_properties ‚Üí tags ‚Üí call-time properties` (a call-time prop overrides a tag overrides a super-prop) ‚Äî realized as binding-merges-tags-then-provider-merges-super (see Technical notes (b)). Named tests required: (1) an off-list TAG key under `on_violation="throw"` RAISES `ValueError`, and under `on_violation="drop-and-error-log"` DROPS the event (no adapter capture) + logs ‚Äî exactly like an off-list super-prop/call-prop; (2) a tag key colliding with a call-time prop ‚Üí the call-time value wins in the delivered event; (3) a tag key that is a declared taxonomy prop with a wrong-typed value is caught by `validate_event_props` post-merge. These pass through the UNMUTATED `provider.capture`, proving the gate/validator crossing without touching the provider.
- [ ] The shipped `provider.py` is UNCHANGED (its `capture` signature + required `distinct_id` untouched); the context read lives in `integrations/context.py`.
- [ ] The context core touches NONE of the PY2-S4 fence's forbidden set: **no `asyncio` module, no event loop, no async client, no `threading`/`queue`** ‚Äî delivery stays sync + thread-offloaded (the offload lives in PY4, not here). `contextvars` is the new stdlib import; `inspect` (`iscoroutinefunction`) and `functools` (`wraps`) are permitted for the async-aware `@scoped` wrapper. The async-aware `@scoped` is context-manager scoping of an async callable ‚Äî NOT async delivery ‚Äî and is outside the fence (architect, 2026-07-10: the fence protects the client's delivery posture, not decorator async-awareness; `integrations/` is a binding-layer module, not a seam/delivery module).
- [ ] `uv run mypy` (strict), `uv run ruff check`, `uv run pytest` all exit 0.

## Technical notes

- **#4 ARCHITECT RULING (2026-07-10, dedicated consult, HIGH confidence all three sub-questions) ‚Äî the load-bearing mechanism:**
  - **(a) `distinct_id` stays REQUIRED on the shipped provider; the context supplies it at the CALL SITE (Option B).** Do NOT widen `capture` to `str | None`. Rejected Option A (widening) because it would either copy PostHog's "personless-UUID" papering (a vendor concept the neutral seam must not adopt ‚Äî `client.py:133`) or masquerade a required arg as optional (a silent-no-identity footgun). The base provider's "server-shaped: per-call distinct_id, no persisted identity" contract is a deliberate seam guarantee ‚Äî Option B preserves it. This maps to the React contract: the React provider does NOT mutate the client's `capture` to be context-aware; the context supplies the client/id and hooks read it at the call site.
  - **(b) Tags merge BEFORE the allowlist gate, final order `super_properties ‚Üí tags ‚Üí call_properties` (later wins).** The final gated bag is `{**super_properties, **tags, **call_properties}`, but the binding does NOT build that whole bag ‚Äî it builds only `{**context_tags, **call_properties}` and passes it as `properties` to the UNMUTATED `provider.capture`, which then runs its existing `_merge_super_properties` (`{**super, **props}`) to prepend the super-props. So: **binding merges tags+call, provider merges super in front** ‚Äî the three-tier precedence is a composition of the two merges, and the tags land in the same bag `provider.capture` already gates + validates. Consumer tags are consumer-supplied ‚Üí **gated** (the E3 rule; a tag bypassing the gate would be a privacy hole). Both references agree on call-time-wins precedence (posthog-python `client.py:162-164` `context_tags.update(properties)`; the shipped `_merge_super_properties` `{**super, **props}`). Because tags land in the merged bag before `provider._allowed` and `validate_event_props`, an off-list tag key is rejected exactly like an off-list call-prop, and a tag whose key collides with a declared event prop IS taxonomy-validated (correct ‚Äî a tag is a property once merged). **Do NOT port posthog-python's flat consumer+computed tag bag** (`django.py:169-220` ‚Äî the `_build_tags` method injecting `$current_url`/`$ip`/`$user_agent`/etc.) ‚Äî keep consumer tags in the gated lane; PY6 has no library-computed lane (see Out of scope).
  - **(c) The context read lives in `integrations/context.py` (+ middleware); `provider.py` stays context-agnostic.** The fence PERMITS `contextvars` in `provider.py` (not in the forbidden set), but the seam provider is the WRONG home: request-scoping is ambient identity, the opposite of the provider's per-call/no-ambient-identity contract; coupling the neutral seam to a framework-binding concern is exactly what `integrations/` (the server analog of the react package) exists to avoid. posthog-python folds context into `Client.capture` because it has a monolithic client with no seam to protect ‚Äî the correct port keeps the `contexts.py` idiom but relocates the CONSUMPTION to the binding layer. Rejected: modifying `provider.py` to read the contextvar (fence-permitted ‚âÝ design-correct).
- **CONTRACT reference (port TO):** `ts/packages/react/src/{analytics-client-provider.tsx,use-analytics.ts,analytics-client-context.ts}` ‚Äî the create-once-client + context-scoped-identity pattern (context supplies, call-site reads, provider surface not mutated), server-shaped as a `contextvars` scope. **DE-BRAND FROM (idiom):** `posthog-python/posthog/contexts.py` (the `ContextScope`+`contextvars` stack + `@scoped`/`new_context()` `@contextmanager` pattern) ‚Äî keep `distinct_id`+`tags`, drop the rest.
- **The `context()` verb realizes the PY2 deferral** ‚Äî PY2's architect deferred `context()` ‚Üí "a request-scoped context manager (see PY6)". This story realizes it as the free `context(analytics) -> ScopedContextView` accessor (PM-locked shape above), server-shaped (`.capture` only, instance-arg not profile-name) ‚Äî deliberately divergent from the TS `context(name) -> ScopedAnalytics` (`.track`/`.page`/`.group`, enrichment-profile) because the browser enrichment semantics are N-A-by-platform (epic Out-of-scope).
