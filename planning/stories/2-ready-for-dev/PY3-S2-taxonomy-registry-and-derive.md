# PY3-S2-taxonomy-registry-and-derive â€” Runtime taxonomy registry + `derive_allowlist_from_taxonomy`

## Why

The runtime taxonomy registry is the full-fidelity half of the two-layer taxonomy: `define_taxonomy(decl)` returns an object whose `.decl` is the walkable registry that both drives `derive_allowlist_from_taxonomy` (the allowlist convenience) and powers runtime prop-type validation. One declaration, two jobs â€” ported from `ts/packages/analytics-kit/src/taxonomy.ts`. It has zero posthog analogue. This story lands the registry + the derive helper (which needs the registry shape, hence it sits after PY3-S1's `enforce_allowlist`).

## Scope

### In

- `analytics_kit/taxonomy.py`:
  - `PropType = Literal["string", "number", "boolean", "date"]` â€” the per-prop type-witness vocabulary (pure data), ported from TS.
  - `PropDecl = dict[str, PropType]` (prop-name â†’ type-tag); `TaxonomyDecl` = a structure carrying `events: dict[str, PropDecl]`, optional `traits: PropDecl`, optional `groups: dict[str, PropDecl]` (a `TypedDict` or dataclass â€” the runtime-walkable decl). **No `page` slot** â€” the TS `TaxonomyDecl` carries an optional `page?: PropDecl` (browser pageview typing), but the server has no pageview surface (`page` is "N-A by platform", `provider.py` docstring), so the `page` slot is a deliberate server omission, not a port miss (record it in a `#`-comment; see Technical notes).
  - `define_taxonomy(decl) -> Taxonomy` â€” returns an instance of a **concrete `Taxonomy` class** (a frozen dataclass or plain class exposing `.decl`), NOT a bare object and NOT a `Protocol`-typed value. The concreteness is load-bearing: `AnalyticsConfig.taxonomy` is typed as `Taxonomy` and Pydantic uses an `isinstance(value, Taxonomy)` guard (see config wiring below), which requires a real runtime class. `.decl` is the registry `derive_allowlist_from_taxonomy` walks and the static layer PY3-S3 brands off. **Reserves NOTHING** (see the architect ruling in Technical notes) â€” no reserved event-name exclusion set, no reserved-key prefix.
  - A **runtime prop-type validator** driven by `.decl`: the `PropType`â†’python-type map (`"string"â†’str`, `"number"â†’int | float`, `"boolean"â†’bool`, `"date"â†’datetime`), used at capture time to validate a consumer's supplied props against their declared types, honoring the `ViolationPolicy` (PY3-S1). Scope the validator to `capture` only, keyed on the event NAME (architect-confirmed 2026-07-10: capture is the one verb where a runtime value â€” the event name â€” selects the prop shape, which is exactly the gap static typing leaves; `set`/`set_group_traits` trait shapes are not name-selected at runtime, so a check there would be net-new surface the epic does not ask for, and traits are already key-gated by S1's allowlist). Two explicit **pass-through** branches (neither raises nor drops): (1) `config.taxonomy` absent â‡’ validator inert; (2) `event` not a declared key in `decl.events` â‡’ out-of-taxonomy, pass through unvalidated (the taxonomy types only what the consumer declares â€” an undeclared event is not a violation). When `event` IS declared, validate the supplied props against that event's `PropDecl`: a declared prop supplied with a wrong runtime type raises/drops per `ViolationPolicy`; a prop absent from the decl, or a declared-but-absent prop, is NOT a type error at this layer (presence/completeness is the static layer's concern, PY3-S3). **Wire it at the provider call-boundary alongside the allowlist gate** â€” run the type check adjacent to the S1 allowlist gate on the capture path, AFTER it (allowlist key-membership is the coarser, cheaper privacy hard-stop; no point type-checking a value not allowed to leave the app). Both guards honor the SAME `ViolationPolicy` instance so the consumer sees one consistent violation posture. Do NOT run the validator for `set`/`set_group_traits` in this story (the derive helper still reads `decl.traits`/`decl.groups`, but runtime type-validation is capture-scoped here; note the boundary in a `#`-comment).
  - `derive_allowlist_from_taxonomy(taxonomy) -> list[str]` â€” a **standalone pure helper** (takes the taxonomy, NOT config): walks `decl.events` prop keys + `decl.traits` keys + `decl.groups` prop keys (the VALUES of events/groups, i.e. each `PropDecl`'s keys â€” so **no event NAMES and no group-TYPE names leak**), returns a **deduped** `list[str]`. Ordering is not part of the contract (the TS returns `[...new Set()]` insertion order; the Python may return any order â€” assert against a `set` in tests, as `allowlist.test.ts` does).
- Config wiring: `AnalyticsConfig` (`config.py`) gains `taxonomy: Taxonomy | None = None` additively (the `define_taxonomy` return type; under `extra="forbid"` it becomes a known field). The `Taxonomy` object is NOT a Pydantic model and must not be coerced/validated field-by-field. Set `model_config = ConfigDict(extra="forbid", arbitrary_types_allowed=True)` AND type the field as the concrete `Taxonomy` class (architect-confirmed 2026-07-10 â€” both, not either): `arbitrary_types_allowed` lets Pydantic hold the opaque object via an `isinstance(value, Taxonomy)` check that passes it through untouched; typing the field as `Taxonomy` (not `Any`, not `SkipValidation`) keeps that free `isinstance` guard, so a consumer who passes a raw dict gets a clear config-boundary error instead of an `AttributeError` on `.decl` deep in capture. `extra="forbid"` (unknown field NAMES) and `arbitrary_types_allowed` (known-field type handling) govern different things and coexist without weakening the forbid-extra posture. The factory threads `config.taxonomy` into the provider (a third construction param, defaulted `None`, mirroring the S1 allowlist threading). **`derive_allowlist_from_taxonomy` is consumer-invoked** â€” the consumer composes it into `config.allowlist` by spread (`allowlist=[*derive_allowlist_from_taxonomy(tax), "super_prop"]`); there is NO library auto-derivation from `config.taxonomy` (see Technical notes â€” the R1 lesson).

### Out

- The best-effort STATIC typing layer (`TypedDict`-per-event, `Literal` event-name union, generic keyed surface, mypy honesty tests) â€” **PY3-S3**. This story is the RUNTIME registry + derive + validator.
- Any reserved event-name set or reserved-key prefix â€” architect-ruled EMPTY/NONE server-side (Technical notes).
- Auto-deriving the allowlist from `config.taxonomy` â€” explicitly NOT done (the R1 bug; Technical notes).
- Server capture/query wiring (PY4/PY5).

## Acceptance criteria

- [ ] `define_taxonomy(decl)` returns an object exposing `.decl` at runtime (the walkable registry).
- [ ] `derive_allowlist_from_taxonomy(tax)` returns exactly the deduped set of all declared event-prop keys + trait keys + group-prop keys; **no event NAMES and no group-TYPE names** appear in the result; a keyless taxonomy derives `[]`.
- [ ] The runtime prop-type validator maps `PropType`â†’python type (`stringâ†’str`, `numberâ†’int|float`, `booleanâ†’bool`, `dateâ†’datetime`) and, on `capture` with a DECLARED event name, a wrong-typed declared prop is caught at runtime honoring `ViolationPolicy` (throw / drop-and-error-log). `bool` is NOT accepted as a `number` (a named test passes `active=True` where `number` is declared and asserts it is rejected â€” the `bool`-is-`int` subclass gotcha; see Technical notes).
- [ ] The validator has two explicit pass-through branches, each with a named test: (1) no `config.taxonomy` â‡’ validator inert, any-typed prop reaches the adapter; (2) `capture` with an UNDECLARED event name â‡’ props pass through unvalidated (out-of-taxonomy is not a violation). The validator is capture-scoped â€” `set`/`set_group_traits` are NOT runtime-type-validated in this story.
- [ ] **`define_taxonomy` reserves NOTHING** â€” a consumer may declare an event named `set_traits` / `set_group_traits` / `group_identify` (they are NOT reserved server-side); no reserved-key prefix is enforced. A dev-only `#`-comment records this is a deliberate server omission (no nameless-fallback mint, no shared super-prop store).
- [ ] **No auto-derivation from `config.taxonomy` (the R1 regression guard):** a named test â€” `test_taxonomy_alone_does_not_activate_allowlist` (or equivalent) â€” builds `create_analytics(config with taxonomy, NO allowlist)` and asserts an off-taxonomy key on a declared event reaches the recording adapter (the allowlist guard stays INACTIVE; typing decision â‰  privacy decision). This is the exact bug TS `node` shipped in R1; the test must be the one that would catch a re-introduction. (Note: the off-taxonomy key must not also trip the prop-type validator â€” use a key absent from the decl, which the validator passes through, OR an undeclared event; the two guards are independent and this test isolates the allowlist one.)
- [ ] `AnalyticsConfig` carries `taxonomy: Taxonomy | None = None` (additive) with `arbitrary_types_allowed=True`; passing a raw dict as `taxonomy` fails config validation at the boundary (the `isinstance` guard), not with an `AttributeError` later. `create_analytics({})` and `create_analytics({"taxonomy": define_taxonomy(...)})` both construct cleanly.
- [ ] **No `page` slot** â€” `TaxonomyDecl` has no `page` field (deliberate server omission; the TS `page?: PropDecl` typing has no server home). A `#`-comment records the omission. The derive helper therefore has no page branch to exclude (unlike the TS derive, which explicitly skips `decl.page`).
- [ ] Composition is consumer-side spread into the single `config.allowlist`; a super-prop key present in no event still passes when spread in; a taxonomy-derived key passes without restating it.
- [ ] `uv run mypy` (strict), `uv run ruff check`, `uv run pytest` all exit 0.
- [ ] Zero vendor token in `taxonomy.py` / docstrings; `grep -ri posthog analytics_kit/taxonomy.py` clean; the library ships zero event names.

## Technical notes

- **CONTRACT reference (ported, not de-branded):** `ts/packages/analytics-kit/src/taxonomy.ts` (`defineTaxonomy` + `PropType`/`PropDecl`/`TaxonomyDecl` + `ShapeOf`) and `ts/packages/analytics-kit/src/allowlist.ts` (`deriveAllowlistFromTaxonomy`), with the pinned name-exclusion + no-auto-derive semantics in `allowlist.test.ts`. **posthog-python has no taxonomy** â€” the library's own surface.
- **Reserved set = EMPTY; reserved prefix = NONE (server-side).** â€” architect (2026-07-10, dedicated consult, high confidence on both):
  - **(1) No reserved event names.** The TS `page`/`pageleave` reservation exists ONLY because those names are minted through a **nameless-fallback / unload path** (`page()` with no arg emits event name `page`; `pageleave` at unload) â€” real name-collision hazards. The server has NO such path (no `page()` verb, no unload). The server internal events (`set_traits`/`set_group_traits`/`group_identify`) are minted by the **verbs** `set()`/`set_group_traits()`, and recognition keys off the structural `internal_kind` discriminant (PY2-S1), NOT the name â€” so a consumer's `capture("set_traits", ...)` is correctly a plain consumer event (`internal_kind=None`). Reserving those names would REINTRODUCE the name-based reasoning the discriminant was built to retire, for zero safety. Rejected: reserving `{set_traits, set_group_traits, group_identify}` â€” solves a collision that can't occur server-side.
  - **(2) No reserved-key prefix.** The TS `__ak_` prefix is browser-persistence-only substrate (`ts/packages/browser/src/persistence-keys.ts`): it keeps the library's internal super-props disjoint from consumer-registered super-props *in a shared persisted store* (the `__ak_groups` case). The server has NO shared super-property store (no `register` verb; super_properties are config-time only, validated at the PY2-S3 config boundary), so there is nothing to namespace-guard. Rejected: porting `__ak_` "for parity" â€” a category error; guards a store the server doesn't have.
  - **Do NOT port** the TS `events: ... & { [K in RESERVED_...]?: never }` construct or any reserved-prefix guard. Leave a dev-only `#`-comment on `define_taxonomy` noting the deliberate server omission (mirrors PY2-S1's documented-omission posture).
- **No `page` slot in `TaxonomyDecl` (deliberate server omission).** The TS `TaxonomyDecl` carries an optional `page?: PropDecl` and its `deriveAllowlistFromTaxonomy` explicitly EXCLUDES page prop keys (there is a dedicated `allowlist.test.ts` case: page props never enter the derived allowlist). The server has no pageview surface (`page` is "N-A by platform", `provider.py` docstring â€” no `page()` verb), so the Python `TaxonomyDecl` omits the `page` slot entirely and the derive helper has no page branch. This is a port omission by design, mirroring the reserved-nothing posture: leave a dev-only `#`-comment on `TaxonomyDecl` noting the deliberate `page`-slot omission so PY8's parity audit reads it as intentional, not a miss. (Same category as PY2-S1's documented N-A-by-platform omissions.)
- **NEVER auto-derive the allowlist from the taxonomy (the R1 hardening lesson).** â€” PM + architect: `derive_allowlist_from_taxonomy` is a **consumer-invoked convenience**, NOT an implicit default. Supplying `config.taxonomy` must NOT activate the allowlist guard. In TS R1, node was the outlier that auto-derived â€” a real bug (taxonomy=typing decision coupled to allowlist=privacy decision, AND it strands super-prop keys that live outside any event's taxonomy). Do NOT reintroduce it. There is ONE config field (`config.allowlist`); composition is the consumer's spread; the guard builds a `frozenset` from that single list. A named test must assert `create_analytics(config with taxonomy, no allowlist)` leaves the guard inactive.
- **`PropType`â†’python map (locked):** `"string"â†’str`, `"number"â†’int | float` (Python splits JS's `number`), `"boolean"â†’bool`, `"date"â†’datetime.datetime`. The runtime validator uses `isinstance` (note `bool` is a subclass of `int` â€” validate `boolean` before `number`, or exclude `bool` from the `number` check, so a `bool` isn't accepted as a `number`). This is a settled map â€” no consult needed.
- **Empty-list edge meets opt-in:** `derive_allowlist_from_taxonomy` returns `[]` for a keyless taxonomy; a consumer spreading only that yields `allowlist=[]`, which under PY3-S1's `is not None` activation is ACTIVE (allow-nothing â†’ everything fails). Intended; call it out in the derive test so the interaction is visible.
- **Neutrality lesson â€” docstrings ship** vendor-neutral; only `#`-comments carry provenance.
