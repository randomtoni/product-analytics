# E12-S1-neutral-flag-port-taxonomy-slot — Neutral flag-port + taxonomy `flags` slot + bootstrap config (both trees, substrate spike)

## Why

No flag adapter can be written until the neutral port is pinned. This spike lands `FeatureFlagPort<TX>` / `FlagSet` / `FlagContext` / the taxonomy `flags` slot / the `flags.bootstrap` config shape **simultaneously** in both `ports` + `taxonomy` files (TS + Python) so the two trees agree exactly before any per-tree adapter work. It is the one place the trees must converge; everything downstream is per-tree adapter implementation against a frozen contract. **MUST be first** — S2–S6 all build against the shape this story pins.

## Scope

### In

- **TS neutral flag port (`ts/packages/analytics-kit/src/ports.ts`)** — replace the `FeatureFlagPort` sketch (`getFlag(key): unknown`) with the async-first snapshot model, generic over the taxonomy shape:
  - `evaluate(context?: FlagContext): Promise<FlagSet<TX>>` — the ONE load-bearing method (async at the boundary; a `refresh?: boolean` option folds `reload` in — see Technical notes for the reload carve-out).
  - `onChange(listener: (set: FlagSet<TX>) => void): () => void` — change-listener returning an unsubscribe; fires-once-with-resolved-set on server adapters, re-fires on reload in the browser (uniform signature, different cardinality).
  - `FeatureFlagPort<TX extends TaxonomyShape = DefaultTaxonomyShape>` — additive generic with the `DefaultTaxonomyShape` default (mirrors `AnalyticsProvider<TX>`); untyped consumers unaffected.
- **TS `FlagSet<TX>` immutable snapshot type** — sync reads off the resolved snapshot: `isEnabled(key): boolean`, `getFlag(key): FlagValue | undefined`, `getPayload(key): PropsOf<TX['flags'][key]['payload']>` (taxonomy-typed), `getAll(): Record<string, FlagValue>`, plus the **neutral degradation signal**: `degraded: boolean` and `reason(key): FlagReason | undefined`. Define `FlagValue = string | boolean` (variant string / boolean) and a neutral `FlagReason` enum-like union (see Technical notes) — vendor eval-quality metadata never reaches this type.
- **TS `FlagContext` type** — `{ distinctId?: string; groups?: Record<string, string>; personProperties?: NeutralProperties; groupProperties?: Record<string, NeutralProperties>; flagKeys?: readonly string[] }`. `distinctId` optional on the type; required-ness on server is adapter-enforced (S3/S4), not a type-level split — the E4 `sessionId` precedent.
- **TS taxonomy `flags` slot (`ts/packages/analytics-kit/src/taxonomy.ts`)** — add a fourth slot to `TaxonomyDecl`: `flags?: Record<string, FlagDecl>` where `FlagDecl = { variants?: readonly string[]; payload?: PropDecl }`. Extend `ShapeOf` / `TaxonomyShape` / `DefaultTaxonomyShape` with a `flags` member so `FlagSet<TX>` reads narrow: `getPayload(key)` → `PropsOf<TX['flags'][key]['payload']>`; `getFlag(key)` narrows to `TX['flags'][key]['variants'][number] | boolean` when variants are declared. Reuse `PropsOf` / `TagToType` verbatim; payloads stay flat-`PropDecl` (nested ⇒ `unknown`).
- **TS bootstrap config (`ts/packages/analytics-kit/src/create-analytics.ts`)** — add `flags?: { bootstrap?: { flags?: Record<string, FlagValue>; payloads?: Record<string, unknown> } }` to `AnalyticsConfig`. Neutral field names (`flags`/`payloads`, never `featureFlags`/`featureFlagPayloads`). Config-shape only in this story — the browser adapter (S2) consumes it.
- **TS index re-exports** — export the new `FlagSet` / `FlagContext` / `FlagValue` / `FlagReason` / `FlagDecl` types from `ts/packages/analytics-kit/src/index.ts` alongside the already-exported `FeatureFlagPort`.
- **Python neutral flag port (`python/src/analytics_kit/ports.py`)** — replace the `FeatureFlagPort` `Protocol` sketch (`get_flag`) with the parity shape: an async-first snapshot `evaluate(context: FlagContext | None = None) -> FlagSet` returning an immutable `FlagSet`, `on_change(listener) -> Callable[[], None]`, and a `FlagSet` with `is_enabled` / `get_flag` / `get_payload` / `get_all` / `degraded` / `reason`. Server-shaped: `distinct_id` required-ness enforced by the adapter (S4). Mirror the runtime-registry + best-effort-static pattern PY3 locked — no attempt at TS compile-time literal typing.
- **Python taxonomy `flags` slot (`python/src/analytics_kit/taxonomy.py`)** — add a `flags` key to `TaxonomyDecl` (a `dict[str, FlagDecl]` runtime shape with `FlagDecl` = variants + payload `PropDecl`), walked by the runtime registry. Do NOT auto-derive an allowlist from flag payloads (flag payloads are inbound, not consumer-supplied outbound props — keep `derive_allowlist_from_taxonomy` untouched).
- **Python bootstrap config (`python/src/analytics_kit/config.py`)** — add the neutral `flags` config field (`bootstrap` with `flags`/`payloads`) to `AnalyticsConfig`, Pydantic-validated, parity with the TS shape.
- **Provider-slot type update (both trees)** — retype the already-declared optional `flags?` slot on `AnalyticsProvider` (TS `analytics-provider.ts:55`) and `AnalyticsProviderImpl.flags` (Python `provider.py:76`) from the sketch port to the new `FeatureFlagPort<TX>` / `FeatureFlagPort | None`. The slot stays optional and `None`/`undefined`-default — NO adapter is wired here. The frozen `AnalyticsProvider` member-count pin must not change (the `flags?` optional member already exists).

## Technical notes

- **Async-first snapshot model — the load-bearing neutrality call (— architect 2026-07-10):** the browser flag read is SYNCHRONOUS in posthog-js (cached in persistence after init fetch); node's is ASYNCHRONOUS (`await evaluateFlags`). Copying the browser's sync shape breaks Python (server-shaped, no persistence/init-fetch) and breaks bar A the moment a self-hosted adapter needs a round-trip. **The neutral read surface is async at the boundary, sync off the snapshot.** PostHog itself converged here — node's per-key `getFeatureFlag`/`isFeatureEnabled`/`getAllFlags` are all `@deprecated` in favor of `evaluateFlags(distinctId, …)` → `FeatureFlagEvaluations` snapshot with sync reads (`posthog-js/packages/node/src/client.ts:1204,1274,1463`; `feature-flag-evaluations.ts`). The neutral port adopts "evaluate once into an immutable snapshot, read synchronously off it." This is a seam decision the PM owns, not a PostHog question — do NOT re-litigate.
- **`reload` carve-out (decide-at-refine, leaning fold-in):** the epic Notes fold `reload` into `evaluate({ refresh?: boolean })` so the port is one async method + a listener + a read-only snapshot. Keep a separate fire-and-forget `reload(): void` ONLY if S2's browser adapter genuinely needs a refresh that returns nothing; default to the folded shape (smaller neutral surface, easier parity + bar A). If a separate `reload` lands, it must land in BOTH `ports` files here in S1 — the trees agree on the port ONCE.
- **`onChange` on a stateless server snapshot (— architect 2026-07-10):** stays on the port, fires-once-with-resolved-set on server (the degenerate case of the browser's re-firing listener — same signature, different cardinality). Rejected: a browser-primary method server adapters omit (an optional-method port weakens the seam + breaks bar-A uniformity). Document the once-semantics as an adapter property (S3/S4 assert it).
- **Neutral degradation signal (— architect 2026-07-10):** `FlagSet` carries `degraded: boolean` + `reason(key): FlagReason | undefined` so a consumer distinguishes "flag is off" from "eval failed / was partial". Without it an eval error collapses silently into a `false`/`undefined` read. The vendor eval-quality metadata that motivates this (`errorsWhileComputing` / `quotaLimited` / `requestId` / PostHog's per-flag `reason` / `locallyEvaluated` on `FeatureFlagEvaluations`) stays **behind the adapter** — only the neutral signal reaches the port. Pick a small neutral `FlagReason` union (e.g. `'resolved' | 'bootstrap' | 'stale' | 'unresolved'`) — name it by consumer-observable state, NOT by vendor field; confirm the exact members at S2 refine once the browser adapter's real states are known, but land the type here.
- **`FlagContext` asymmetry (— architect 2026-07-10):** `distinctId` optional on the TYPE; **required on the server adapter** (validated — no ambient actor, mirrors node `evaluateFlags(distinctId, …)`), **optional on the browser adapter** (filled from current identity). Same interface, different required-ness enforced by the adapter — exactly how E4 handled `sessionId` (browser stamps it, node leaves it unset). Do NOT invent a fake ambient server actor to force identical signatures.
- **Local-vs-remote is adapter-internal (— architect 2026-07-10):** never a port parameter. `onlyEvaluateLocally`/`strictLocalEvaluation`/poll-interval are adapter config (E13), read by node/Python, ignored by browser. The consumer asks the same question either way; where it's answered is an adapter strategy (exactly how the query client hides HTTP-vs-warehouse behind one `funnel()`).
- **Bootstrap = config, not a method (— architect 2026-07-10):** server-rendered data handed to the client at construction, available synchronously at init before any method could run — a method inverts the timing. Neutral field names strip PostHog's `featureFlag*` prefix. Browser-*primary* (kills a first-paint flash that only exists client-side) but the config field is neutral and Python accepts it too (SSR request-scoped eval) — advances BOTH trees, NOT Python-N/A. The adapter maps it into its backend seeding (browser reference: `posthog-js/packages/browser/src/posthog-featureflags.ts:339` `initialize()`); consumption is S2.
- **Taxonomy `flags` slot (— architect 2026-07-10):** PostHog has no compile-time taxonomy — payloads are untyped `JsonType`, so `defineTaxonomy` is a library-original shape (verified against `ts/packages/analytics-kit/src/taxonomy.ts`: `PropType`/`PropDecl`/`PropsOf`, slots `events`/`traits`/`groups`/`page`). Add `flags` as the fourth slot; reuse `PropsOf` / `TagToType` verbatim. Payload nesting stays flat for v1 (`PropDecl`); nested ⇒ `unknown` — the same pragmatic ceiling `PropsOf` already has. PostHog types `getFlagPayload` as loose `JsonType | undefined`, so a recursive JSON-schema `PropType` is real complexity with no reference driver — REJECTED for v1, noted as a taxonomy-hardening follow-up orthogonal to the FF port. Python mirrors via the PY3 runtime-registry + best-effort-static pattern (a `flags` registry entry, boundary-validated, statically hinted where the checker follows) — the same "not the TS compile-time literal guarantee" ceiling PY3 locked.
- **Carry `TX` through the port generically:** `flags?: FeatureFlagPort<TX>` on `AnalyticsProvider<TX>` — additive generic with `DefaultTaxonomyShape` default, exactly the shipped `AnalyticsProvider<TX = DefaultTaxonomyShape>` pattern. Untyped consumers unaffected.
- **Coordination boundary (— architect 2026-07-10):** the neutral port + types are decided ONCE, here, in BOTH `ports`+`taxonomy`+`config` files simultaneously, before any adapter work. This is the one place the trees must agree exactly. Do NOT split this epic by language. If S2–S4 discover the port shape is wrong, the fix comes back HERE and re-propagates to both trees — never a per-tree drift.
- reference: `posthog-js/packages/core` (shared flag primitives) + `packages/node/src/feature-flag-evaluations.ts` (the snapshot shape to neutralize); de-brand.
