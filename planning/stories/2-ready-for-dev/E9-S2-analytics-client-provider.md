# E9-S2-analytics-client-provider ‚Äî SSR-safe `AnalyticsClientProvider` + context

## Why

The core of the binding: a React context provider that places the neutral client in context so a consumer wraps their tree once instead of hand-threading the client through every component. It must be SSR-safe (Next.js) and StrictMode-safe, and it must construct the client via the inward `@analytics-kit/browser` path when given a `config`.

## Scope

### In

- **`AnalyticsClientContext`** ‚Äî a React `createContext` whose **default value is a distinct module-private sentinel** (a unique `Symbol` or a unique frozen object), NOT `undefined`. The sentinel encodes exactly ONE binary: "no provider anywhere above me" (context still holds the sentinel ‚Üí S3's hook throws) vs "inside a provider" (context holds a **real `RootAnalytics<TX>`**, always ‚Äî synchronous construction means there is NO not-ready/`undefined` window; see the synchronous-construction note below). So the context value type is `RootAnalytics<TX> | typeof NOT_IN_PROVIDER_SENTINEL` (or `{ client: RootAnalytics<TX> } | sentinel` if a wrapper object is preferred ‚Äî but there is no `client: undefined` state). Internal module (`src/context.ts` or `src/analytics-client-context.ts`); the context object may be exported for advanced use but the hook (S3) is the intended read path.
- **`AnalyticsClientProvider`** component (named by role ‚Äî see naming note) accepting **discriminated-union props**, mutually exclusive:
  - `{ client: RootAnalytics<TX>; config?: never }` ‚Äî use the caller's pre-built client as-is.
  - `{ config: AnalyticsConfig; client?: never }` ‚Äî construct a browser client internally via `@analytics-kit/browser`'s `createAnalytics(config)`. (Do NOT write `AnalyticsConfig & { taxonomy? }` ‚Äî `AnalyticsConfig` **already declares** an optional `taxonomy?: Taxonomy<TaxonomyDecl>` field (`create-analytics.ts:43`); the intersection is redundant and would shadow the real field type.)
  - Plus `children`.
- **Client-wins-with-warning** when both are somehow passed (defeating the type via `any`): the `client` prop wins; emit a dev-mode `console.warn` (guarded so it's silent in production ‚Äî gate on `process.env.NODE_ENV !== 'production'`).
- **Synchronous, create-once construction (SSR-safe)**: the `config`-branch client is constructed **synchronously via a `useState` lazy initializer** ‚Äî `const [client] = useState(() => createAnalytics(config))` ‚Äî so the client exists in context at **render 1**, on the server and the client alike, with NO first-render `undefined` window. This is SSR-safe because `createAnalytics(config)` is **DOM-free at construction** (verified ‚Äî see the construction-is-DOM-free note); the render-time no-DOM contract is what makes synchronous construction safe (the E6 `typeof window`/`document` guards handle it). Deliberately NOT effect-only: the epic's earlier "never render/useMemo" lock existed to keep DOM access out of render for SSR, but construction touches no DOM, so that rationale is void ‚Äî posthog-js itself constructs synchronously (`PostHogProvider.tsx:49` `useMemo`) and defers only the DOM-touching `.init()`. The context always holds a real `RootAnalytics<TX>`. (`useState` lazy init, not `useMemo`: `useMemo` is not a semantic create-once guarantee in React ‚Äî it may discard and recompute; `useState`-init encodes "exactly one client per provider mount".)
- **StrictMode + create-once**: a `useState` lazy initializer runs exactly once per mount, so StrictMode's dev double-invoke does NOT double-construct within a single mount. StrictMode's simulate-unmount/remount will construct ‚Üí `shutdown()` (config-branch cleanup) ‚Üí reconstruct ‚Äî which is fine: each construct is a fresh DOM-free SSR-safe build, and the interleaved `shutdown()` drains nothing meaningful (no DOM/transport work has happened pre-mount effects). No separate ref-guard is needed for the double-*construction* case ‚Äî the lazy `useState` is the guard. When only `client` is passed, no construction happens at all.
- **No auto-pageview**: the provider wires context ONLY. It captures NO pageviews and installs NO history listener (BRIEF ¬ß1; epic Notes architect 2026-07-07). `page()` stays manual/router-driven ‚Äî the optional router hook is S4.
- **Whole-stack no-op flows through**: when `config` is unkeyed, `createAnalytics` already returns the no-op-backed client (browser `resolveAdapter` ‚Üí `NoopAdapter`); the provider builds and provides it identically. The provider itself adds no keyed/unkeyed branching ‚Äî the no-op posture rides the client (bar B).
- **Export the provider from `src/index.ts`**: this story owns exporting `AnalyticsClientProvider` (and the props type) from the package entry ‚Äî replacing the `version`-only stub export. `AnalyticsClientContext` MAY be exported for advanced use. S3 adds the `useAnalytics` export; S4 adds `usePageView`. (Under the strict S1‚ÜíS2‚ÜíS3‚ÜíS4 order, S2 is the first story to ship a real public symbol.)

### Out

- `useAnalytics()` hook ‚Äî S3 (this story ships the context + provider; a temporary `useContext` in the smoke test is fine but the exported hook lands in S3).
- `usePageView()` / any router integration ‚Äî S4.
- Feature-flag / session-replay React helpers (typed extension points only this release ‚Äî epic Out of scope).
- Reacting to `config` changes after mount beyond the StrictMode guard (posthog-js re-runs `set_config` on options change; NOT in scope ‚Äî a changed `config`/`client` prop after mount is a consumer footgun; document that the client is constructed once, mirror is deferred to the expansion path).

## Acceptance criteria

- [ ] `<AnalyticsClientProvider config={...}>{children}</AnalyticsClientProvider>` renders children and provides a constructed `RootAnalytics` via context **from render 1** (synchronous `useState` lazy init ‚Äî no post-mount wait, no `undefined` window).
- [ ] `<AnalyticsClientProvider client={existing}>` provides the caller's client without constructing a new one.
- [ ] **SSR-safe**: rendering the provider under an SSR pass (no DOM / `renderToString`-style) constructs the client synchronously WITHOUT any DOM/`window`/`document` access throwing ‚Äî construction is DOM-free (relies on the E6 guards). The client is available in context during that render; a child's `useAnalytics()` sees a real client, not `undefined`.
- [ ] **StrictMode**: the `useState` lazy initializer constructs the client exactly once per mount (a test wrapping the provider in `<React.StrictMode>` asserts one construction within a mount; the simulated unmount/remount may construct‚Üíshutdown‚Üíreconstruct, which is acceptable and must not leak or double-provide).
- [ ] Passing both `client` and `config` (via cast) makes `client` win and emits exactly one dev-mode warning; the warning is suppressed when `NODE_ENV === 'production'`.
- [ ] The provider auto-captures no pageviews (no `page()` call and no history listener installed by mounting the provider) ‚Äî asserted with a recording/mock client.
- [ ] Unkeyed `config` ‚Üí the provider still renders and provides the no-op-backed client (bar B); `track`/`page` through it are silent no-ops.
- [ ] Zero vendor references; the provider calls only `@analytics-kit/browser`'s public `createAnalytics` ‚Äî never an adapter internal (bar A seam intact).
- [ ] Frozen-15 held: the provider exposes the client's existing facade + `context()`; it adds NO new facade verb.
- [ ] `AnalyticsClientContext`'s createContext DEFAULT is a distinct module-private sentinel (NOT `undefined`), so S3's hook can tell "no provider" (sentinel ‚Üí throw) from "inside a provider" (a real `RootAnalytics`). A test reads the raw context outside any provider and asserts it is the sentinel, and inside a provider asserts it is a real client (never `undefined` / never not-ready).

## Technical notes

- **Naming ‚Äî resolved (architect 2026-07-08):** the React context component is **`AnalyticsClientProvider`** (named by role ‚Äî "provides the analytics client to the tree"), the context is **`AnalyticsClientContext`**. This deliberately does NOT reuse the seam's pinned TS interface name `AnalyticsProvider` (which is the 15-member client *contract*, unrenameable). Rationale: posthog-js can call theirs `PostHogProvider` because their interface is `PostHog`, not `PostHogProvider` ‚Äî copying that pattern here would collide with our `AnalyticsProvider` interface. `...Provider` keeps the React ecosystem affordance; `Client` disambiguates from the interface. De-brands `posthog-js/packages/react/src/context/PostHogProvider.tsx` + `PostHogContext.ts` (shape reference only; posthog-specific bits ‚Äî `getDefaultPostHogInstance`, `window.posthog` fallback, `bootstrap` ‚Äî are dropped).
- **Discriminated-union props** ‚Äî epic Notes (architect 2026-07-07): existing `client` XOR `config`, mutually exclusive; client wins when both passed, with a dev-mode warning. Reference shape: `PostHogProvider.tsx:23-25` (the union) and `:52-60` (the both-passed warnings). Our discriminant is `client` vs `config` (posthog's is `client` vs `apiKey`).
- **Synchronous create-once construction, SSR-safe** ‚Äî orchestrator decision (2026-07-08, relaxing the epic's earlier "effect-only, never render/useMemo" lock): the `config`-branch client is constructed **synchronously in render via a `useState` lazy initializer** ‚Äî `const [client] = useState(() => createAnalytics(config))`. It exists in context at render 1 on server and client, so there is NO not-ready/`undefined` window and the hook's return contract stays a clean `RootAnalytics<TX>`. Why the lock was relaxed: it existed to keep DOM access out of render for SSR, but `createAnalytics(config)` is **DOM-free at construction** (verified ‚Äî see the construction-is-DOM-free note), so the SSR rationale is void. This aligns with posthog-js, which constructs synchronously (`PostHogProvider.tsx:49` `useMemo` returns an instance immediately) and defers ONLY the DOM-touching `.init()` to an effect ‚Äî we have no separate DOM-touching init step, so nothing needs deferring. Use `useState` lazy init, NOT `useMemo`: `useMemo` may legitimately discard and recompute (not a create-once guarantee); `useState`-init constructs exactly once per mount. Any FUTURE DOM-requiring post-mount work (there is none today) would go in a `useEffect`, but it must not gate client availability.
- **Client ownership + cleanup asymmetry.** The `config` branch OWNS the client it constructs: its effect cleanup (on unmount) should call `client.shutdown()` (drains the in-flight buffer, detaches listeners ‚Äî `browser-adapter.ts` `shutdown`). The `client` branch must NOT shut down a client it did not create ‚Äî ownership follows construction. Encode this asymmetry so an unmount doesn't tear down a caller-owned client.
- **Construction-is-DOM-free is a contract, not an incidental fact.** Verified (architect 2026-07-08): `createAnalytics(config)` ‚Üí `resolveAdapter` ‚Üí `BrowserAdapter` constructor touches NO `window`/`document`/`localStorage` unguarded ‚Äî every DOM access is `typeof ‚Ä¶ === 'undefined'`-guarded and degrades to memory/no-op under SSR (`storage-backends.ts`, `browser-adapter.ts` `bindUnloadListeners`/`bindAutocapture` are SSR-guarded). This is precisely what makes **synchronous render-time construction** safe (and is why the effect-only lock could be relaxed). Treat "every target adapter's `createAnalytics` is DOM-free at construction" as a REQUIREMENT the React binding's SSR-safety depends on ‚Äî not a browser-adapter internal to lean on silently. If a future adapter's `createAnalytics` ever touched the DOM at construction, this synchronous posture would break and construction would have to move behind a guard.
- **StrictMode / create-once ‚Äî the `useState` lazy init IS the guard.** No separate `useRef<PreviousInitialization>` guard is needed for double-*construction*: `useState(() => createAnalytics(config))` runs its initializer exactly once per mount, so StrictMode's dev double-invoke does not build two clients within one mount (this replaces posthog's ref-tracking approach `PostHogProvider.tsx:47,:91-107`, which they need only because they support live apiKey/options mutation ‚Äî we do not). StrictMode's simulate-unmount/remount WILL construct ‚Üí `shutdown()` ‚Üí reconstruct across the remount; that is acceptable (fresh DOM-free build each time; the interposed `shutdown()` drains nothing meaningful pre-DOM). The client is constructed once from the INITIAL `config`; a later `config` prop change does NOT reconstruct (create-once semantics ‚Äî a changed `config` after mount is a documented consumer footgun, per Scope.Out; a consumer needing to swap config should remount or manage the `client` prop themselves). Because construction ignores later `config` identity, the stable-`config`-reference memoization posthog documents (`:79`) is NOT required here.
- **Inward construction** ‚Äî the `config` branch calls `@analytics-kit/browser`'s `createAnalytics` (public factory) ‚Äî the browser target is the peer dep (S1). Dependencies point inward: `react ‚Üí browser ‚Üí seam` (epic Notes architect ¬ßE1). Never import a target adapter into the react package or the seam.
- **Exact imports (pinned).** The **runtime** `createAnalytics` MUST be imported from `@analytics-kit/browser` (the construction path is the browser target). The **types** `RootAnalytics` and `AnalyticsConfig` are re-exported by BOTH `@analytics-kit/browser` and `analytics-kit`; import them alongside `createAnalytics` from `@analytics-kit/browser` for locality (one import line for the construction concern). If S2 threads the taxonomy generic (`TX`), the taxonomy types (`TaxonomyShape`/`DefaultTaxonomyShape`) are NOT on `@analytics-kit/browser` ‚Äî import those from `analytics-kit` (see S1's two-deps note). Type the `config` prop as `AnalyticsConfig` (the default, non-generic shape); full taxonomy-through-props generics stay in S3 where the hook return type carries `TX`. Keep S2 typed to the default shape ‚Äî the generic is not trivial to thread through a discriminated union and is unnecessary for the provider's job (it stores whatever `RootAnalytics<TX>` it built or received).
- **No auto-pageview** ‚Äî `PostHogProvider.tsx:140` (the return wires context and does NOT capture) is the reference; ours holds the same posture (BRIEF ¬ß1).
- Do NOT expose `AnalyticsProviderImpl` or reach into the client's internals ‚Äî the provider is a pure consumer of the public `RootAnalytics` facade.
