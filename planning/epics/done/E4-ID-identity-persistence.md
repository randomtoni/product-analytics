# E4-ID-identity-persistence — Browser identity, persistence & the shared browser substrate

## Why

Every browser event needs a stable actor to attach to before the consumer ever calls `identify()`, and that anonymous identity must survive reloads and stitch across subdomains for pre-login funnels. This is the first target-package epic and the substrate the whole `identify` area is stabilized around; it also lands the persistence layer that the `capture` cycle (E5/E6) builds directly on. **It fills the hooks E2/E3 deliberately left as stubs** — the facade's `currentDistinctId()` accessor, the empty `reset()` body, the in-memory-only `optedOut` state, the never-set `NeutralEvent.sessionId`, and the `getPersistedProperty`/`setPersistedProperty`/`alias` SPI verbs that shipped ready but unused. Informed by `research/ARCHITECT-RELEASE1.md` §E4 and §E-cross; refreshed against the shipped `core` seam and the E2/E3 review forward-notes.

## Success criteria

- An anonymous distinct id (UUIDv7) is generated at first load, persisted, and reused across reloads; a **device id** is minted once at first load as a **separate persisted identifier** (survives `reset()` unless `resetDevice`). Identity state is modeled as an explicit neutral `anonymous | identified` value **inside the adapter** (never the wire id-equality trick); E4 keeps it adapter-internal — the only public identity accessor is `currentDistinctId()`, no public identity-state getter yet (— architect 2026-07-08 refiner pass, open-Q closed). The resolver replaces the facade's stubbed `currentDistinctId()` via a new SPI `getDistinctId()` — pure population, no `NeutralEvent` reshape.
- Persistence mode is config-selectable (`cookie` | `localStorage+cookie` | `memory`), defaulting to `localStorage+cookie`; the cookie half carries only the small identity/session keys. The store lives entirely inside the browser adapter behind `getPersistedProperty`/`setPersistedProperty`.
- A config-supplied `cookieDomain` is authoritative for cross-subdomain sharing; the public-suffix auto-probe is used only as fallback. A simulated cross-subdomain journey keeps one distinct id.
- `identify(id, traits, traitsOnce)` performs the anonymous→identified merge **client-side only**, and only when the id differs AND the actor is still anonymous; re-identifying with the same id updates traits; an already-identified actor identifying with a new id does not merge. The merge rides the SPI `identify()` verb (not `alias`); the guard lives in the browser adapter.
- `traits` (mutable) and `traitsOnce` (first-touch-immutable) follow register / register_once semantics.
- Super-properties are registered via `register` / `registerOnce` / `unregister`; **registered keys are gated by the E3 allowlist at registration time** (facade `allowed()`), so consumer PII carried in a super-prop cannot bypass the privacy contract. The adapter's downstream merge-into-events is trusted because everything in super-prop storage already passed the gate.
- A session id (UUIDv7) is assigned and expires on idle (30 min default) or max length (24 h default), both configurable; the browser adapter stamps `NeutralEvent.sessionId` in its capture pipeline (the facade leaves it undefined). A session id is minted even in `persistence: 'memory'` mode.
- `reset()` regenerates the anonymous id and clears identity + persistence + session, but keeps the device id unless an explicit `resetDevice` flag is passed. It is a new SPI `reset(options?)` verb that stays effective even while opted out (routes to the retained live adapter).
- The opt-out **decision is durably persisted** (survives reload) and read at browser-adapter construction **before** any cookie/domain-probe write — so an opted-out or unkeyed client writes zero cookies. This makes the whole-stack no-op E2 promised real via the persistence SPI; `flush()`/`shutdown()`/`reset()` remain effective under opt-out.
- **Bar B:** cookie domain, scope, persistence mode, and consent default are config-only; the library hardcodes none. **Zero vendor references** in storage key names, state names, or the neutral surface.

- The `optIn` / `optOut` / `hasOptedOut` **interface** and the config-selected no-op factory — **E2** (this epic adds the durable-persistence half and the lifecycle-vs-consent routing).
- Server-side identity merge (identified→identified) — not release 1; PostHog does anon→identified client-side only.
- `alias()` implementation. The `alias(previousId, distinctId)` verb already ships on the SPI (E2), but the anon→identified merge rides `identify()`, **not** `alias` — so E4 adds NO facade `alias()` method and NO adapter alias implementation. Aliasing two known ids is a later slice.
- `group()` typing and event-time super-property application at capture — thread through `defineTaxonomy` (E3) and the capture path (**E6**).
- **Cross-tab session/persistence synchronization** (PostHog's cross-tab activity-persist + session hand-off) — explicitly deferred, not silently dropped. E4 lands single-context persistence + session id; cross-tab sync is a later hardening slice.

## Notes

### Standing mechanics (— architect 2026-07-07; still valid post-ship)

- **First story is the shared-decomposition spike, not an identity feature.** posthog-js's browser core is one ~4,200-line monolith (`posthog-core.ts`) and E4/E5/E6 all slice the SAME file; the real cost is de-branding/de-coupling it into the neutral facade + adapter, not the per-feature logic. Front-load the E4/E5/E6 shared substrate (neutral event object + property-build order + persistence layer) before the tracks split.
- **Default persistence is `localStorage+cookie`, not pure cookie.** PostHog abandoned the pure-cookie default (~4 KB cap) — its own default is `'localStorage+cookie'` with the source comment "up to 1.92.0 this was 'cookie'" (`posthog-core.ts:235`, refiner-verified); the cookie half carries only identity/session keys for cross-subdomain sharing, localStorage holds the bulk. The BRIEF's "cookie default" is satisfied by the cookie-backed identity keys — a PM-confirmed default choice, not a technical unknown.
- **Cross-subdomain: explicit `cookieDomain` authoritative, auto-probe is fallback.** The eTLD public-suffix probe (throwaway `dmn_chk_*` cookies) is fragile; the consumer supplies the domain per BRIEF, the probe only de-risks the missing-config case.
- **Anonymity is an explicit neutral state; the wire encoding stays in the adapter.** The neutral surface carries `anonymous | identified`; the `distinct_id === $device_id` trick, the `$sesid` `[lastActivity, id, start]` tuple, `$anon_distinct_id`, and `ph_`-prefixed names are all [WIRE], normalized inside the browser adapter. No `$`-prefixed names on the neutral surface.
- **identify() merge guards (client-side only).** Merge only when the new id differs AND the actor is still anonymous; same-id re-identify just updates traits; already-identified + new id does NOT merge client-side. No server-side identity merge this release.
- **Session defaults: idle 30 min, max 24 h, UUIDv7; must mint even in memory mode.** Sessions are needed by E6 enrichment and transport session-props, so `persistence: 'memory'` must still produce a session id or E6 breaks. Sessions are independent of replay (out of scope).
- **reset() keeps the device id by default.** Regenerate the anonymous id + clear state; regenerate the device id only on an explicit `resetDevice` flag (PostHog's default keeps it).

### Seam placement — locked against the shipped surface (— architect 2026-07-08)

- **Identity state lives in the browser adapter; the facade stays a thin delegator (Q1).** Add SPI `getDistinctId(): string` (contract: cheap synchronous in-memory read — the adapter loads persistence once at init and caches, it does NOT hit storage per event). Facade `currentDistinctId()` becomes `return this.liveAdapter.getDistinctId()` — exactly the one-method swap the E2 consolidation anticipated. Move the `'anonymous'` constant down to `NoopAdapter.getDistinctId()` so "unkeyed ⇒ anonymous" stays structural. Read from `liveAdapter` (not the consent-swapped `adapter`): identity is orthogonal to consent, so the distinct id stays truthful while opted out. **Rejected:** facade orchestrating the resolver via `getPersistedProperty('distinct_id')` — leaks the [WIRE] key name into the neutral facade and dictates persistence-key semantics to every adapter (bar-A break).
- **The anon→identified merge rides `identify()`, not `alias()`; the guard lives in the browser adapter (Q2).** Source-confirmed (refiner-verified against the checkout): PostHog emits the merge inside `identify()` as a capture carrying `$anon_distinct_id` (browser `posthog-core.ts:2552-2570`, the `$anon_distinct_id` stamp at `:2570`; the prior-id link is set in core `posthog-core.ts:363,374`); `alias` is an unrelated link-two-known-ids op. The facade's `identify()` keeps delegating straight to `adapter.identify(...)`; the adapter grows the de-branded guard (differs+anonymous ⇒ merge; same id ⇒ traits-only; already-identified+new id ⇒ no merge). **Cross-dependency:** the facade's `setTraits` calls `adapter.identify(currentDistinctId(), ...)`, so its correctness is gated on Q1's real resolver shipping first — with the stub returning `'anonymous'`, `setTraits` could mis-trigger a merge.
- **The browser adapter stamps `sessionId`; the facade leaves it undefined (Q3).** The session id comes from a stateful `checkAndGetSessionAndWindowId(timestamp)` call that advances the idle clock and mints on expiry — behavior, not a storage read, and browser-only (node has no sessions). `NeutralEvent.sessionId` is optional precisely because it's adapter-populated (node leaves it unset). No new SPI accessor and it must NOT ride `getPersistedProperty` (a KV read can't express "give me the current id, advancing idle from this timestamp"). **Nuance:** "session minted even in memory mode" means the real browser adapter with `persistence: 'memory'` (mint is independent of storage backing) — NOT the whole-stack `NoopAdapter`, where every event is dropped so an unset `sessionId` is harmless.
- **reset() is a new SPI verb routed to the live adapter (Q4).** Add SPI `reset(options?: { resetDevice?: boolean })`; the facade's empty `reset()` delegates. Do NOT orchestrate reset from the facade via `getPersistedProperty`/`setPersistedProperty` — reset is a multi-step transition against [WIRE] keys + stateful managers (`persistence.clear()`, state→anonymous, `register_once` a fresh distinct id keeping `$device_id`, `resetSessionId()`), and facade-side KV would leak `distinct_id`/`$device_id`/`$sesid` into the neutral layer. Widen the consumer `reset()` → `reset(options?)` (additive/optional). **Route reset to `liveAdapter`, not the consent-swapped `adapter`:** a logout during opt-out must still clear identity (routing to the no-op would leave stale identity — a privacy footgun); the live adapter's own consent posture keeps persistence suppressed, so no cookie is written.
- **Durable consent is adapter-owned and read first at construction (Q5a).** Consent durability is storage, exposed via an SPI pair `getConsentState()` / `setConsentState(state)` (signature now locked to the tri-state enum — see the sub-bullet below). The browser adapter reads consent FIRST at its own construction, from a dedicated side-effect-free read, **before** building the cookie/domain-probing persistence — so an opted-out/default-denied client writes zero cookies (the domain-probe itself sets throwaway `dmn_chk_*` cookies, so it must be gated). The facade cannot own this guarantee: `create-analytics` hands it an already-constructed adapter, by which point the probe may have run. The facade seeds its in-memory `optedOut` from `adapter.getConsentState()` at startup and delegates `optOut`/`optIn` to `setConsentState` for durability, keeping the no-op swap as the fast runtime capture gate. This consent read must be **separate** from the main persistence (reading consent through the persistence you're gating is circular).
  - **Consent state is a tri-state neutral enum, not a boolean (— architect 2026-07-08 refiner pass, open-Q closed, high confidence).** `type ConsentState = 'granted' | 'denied' | 'pending'` — neutral string literals, never PostHog's numeric `{-1, 0, 1}`. A boolean cannot separate "explicitly denied" from "not yet asked", and that distinction is the hinge Q5 depends on: only `'granted'` writes cookies, so `'pending'` (default/unasked) must write **zero** cookies exactly like denied — a boolean collapses that. `'pending'` resolves to a runtime capture decision against the **config consent-default** (the opt-in-by-default vs opt-out-by-default field E4 adds — the default deferred from E2-S5): `facade.optedOut = (resolved === 'denied')`. **PostHog folds `doNotTrack` into denied** (`consent.ts:29-31,122-131`); the browser adapter likewise resolves any platform DNT/GPC signal **into** `getConsentState()` so the neutral seam sees ONE resolved value — DNT is [WIRE]/platform mechanics normalized inside the adapter, never a neutral-surface concept. **Rejected:** boolean consent — cannot express the pending state that gates whether any storage write happens.
- **Lifecycle routes to the live adapter; opt-out drops, not flushes; one derivation method (Q5b).** `flush()`/`shutdown()` change from `this.adapter.*` to `this.liveAdapter.*` — they are lifecycle ops on real resources (batching timer, beforeunload listener, in-flight queue) only the live adapter holds; routing them through the consent-swapped no-op leaks the live adapter's timer/listener on an opted-out shutdown (the reviewer-flagged bug, E2-S5). `optOut()` quiesces the live adapter via `setConsentState('denied')` (stop timer + listener, **drop** the unsent buffer — flushing would send data the user just declined) and swaps the active delegate to the no-op as defense-in-depth. **The single swap method:** make `this.adapter` a pure derivation of (`liveAdapter`, `optedOut`) written only through one private `resyncActiveAdapter()`; a post-construction `installAdapter(next)` sets `liveAdapter` then re-derives — so `optOut()→optIn()` can never restore a stale ref (opt-in re-derives from the *current* live adapter, never a snapshot). **Locked routing invariant:** capture / identity-mutation ops (`track`, `page`, `identify`, `group`, `setTraits`, `register*`) route through `this.adapter` (consent-swappable ⇒ inert under opt-out, no persistence writes); identity-read (`getDistinctId`), lifecycle (`flush`, `shutdown`), and `reset` route through `this.liveAdapter`. Consolidate the field-derivation **before** adding `installAdapter`, or the stale-ref bug becomes real.
- **Super-props are gated at registration in the facade, never auto-trusted downstream (Q6).** E4 adds facade `register(props)` / `registerOnce(props)` / `unregister(key)` (+ SPI verbs); each consumer-supplied-key method runs the E3 `allowed()` guard on the incoming keys **before** persisting, then delegates. Because everything in super-prop storage already passed the gate on the way in, the adapter's downstream merge-into-events stays ungated/trusted — the "downstream = trusted" exemption applies ONLY to library-computed keys (session id, enrichment), never to consumer-supplied super-props. Failing loud at the `register()` call names the exact PII source (best DX). Reuses the single `config.allowlist` set (E3): a super-prop key not on the allowlist ⇒ `register()` throws (default) or drops-and-logs — consistent with E3 already requiring the consumer to spread super-prop keys into `config.allowlist` (they live outside any event's taxonomy entry). Under opt-out, `register*` routes through the no-op `adapter` ⇒ inert, no persistence write. **Rejected:** merging super-props downstream as library-trusted — silently auto-trusts consumer PII, the exact failure to avoid.
- **No double-gate, no gap — composition verified against the E3 guard (— architect 2026-07-08 refiner pass, high confidence).** Every consumer-supplied key crosses `allowed()` exactly once: super-props at `register*`, per-call props/traits at `track`/`page`/`identify`/`group`/`setTraits` (the shipped E3 guard). The adapter's super-prop merge-into-events happens **below** the facade guard, so a stored super-prop is never re-gated — no double-gate. `register*` gates the incoming bag before persisting — no gap. `register*` MUST reuse the shipped `allowed()` whole-bag drop/throw semantics verbatim (first off-list key ⇒ throw by default, or drop-and-error-log) — same policy path as `track`, not a new one.
- **Library-computed identifiers leave the app by a DELIBERATE, documented exemption (— architect 2026-07-08 refiner pass).** `sessionId`, `device id`, library id, and future enrichment are stamped downstream (below the guard) and so bypass the consumer allowlist. The BRIEF's allowlist governs "properties permitted to leave the app" — and these identifiers **do** leave the app — so this must be stated as an intentional exemption, not left implicit: the allowlist gates **consumer-supplied** payload (potential PII); library-computed identifiers are not consumer PII and are trusted by construction. Consumer super-props are the ONE consumer-supplied source that flows downstream into every event, which is exactly why they alone are gated at `register()` — never auto-trusted.

### Additive SPI delta (fold into stories; all trivially satisfiable by `NoopAdapter` + node)

- **`api_impact: additive` is scoped to the CONSUMER API, and bar A holds (— architect 2026-07-08 refiner pass, medium-high).** The public `AnalyticsProvider` facade + `AnalyticsConfig` grow only optionally (`reset()`→`reset(options?)` is additive; new `register*` methods; new optional config fields), so provider-swap stays one adapter with zero consumer change (bar A). The **SPI (`AnalyticsAdapter`) grows these as REQUIRED verbs** — strictly an SPI-contract expansion, i.e. breaking to any *external* adapter author, harmless today only because the sole implementors are in-repo (`NoopAdapter` + the future node/browser targets, all internal modules per CLAUDE.md) and are updated in lockstep. Read: **additive (consumer API) / SPI-expanding (no external implementors pre-1.0)**. All verbs are trivially no-op/node-satisfiable.
- `getDistinctId(): string` — Q1 (cheap in-memory read). `NoopAdapter.getDistinctId()` returns the `'anonymous'` constant (moved down from the facade).
- `reset(options?: { resetDevice?: boolean }): void` — Q4.
- `register` / `registerOnce` / `unregister(key)` — Q6 (once-semantics + merge-on-capture is stateful behavior, not raw KV — do NOT ride `setPersistedProperty`). **One open sub-question, non-blocking, decided at the super-prop story's refine (— architect 2026-07-08 refiner pass, leans (ii) at medium-high):** the once-semantics API shape — (i) separate `register(props)` / `registerOnce(props)` (mirrors PostHog SOTA + the shipped `identify(id, traits, traitsOnce)` two-bag idiom) vs (ii) collapsed `register(props, { once? })` (mirrors the shipped `setTraits(traits, once?)` flag idiom, one "once" idiom library-wide). Does NOT block the substrate spike or persistence stories. `unregister` stays a distinct inverse either way.
- `getConsentState(): ConsentState` / `setConsentState(state: ConsentState)` — Q5, **signature locked to the tri-state `'granted' | 'denied' | 'pending'`** (see Notes Q5). Kept a get/set **pair** (a single overloaded getter/setter is a JS anti-pattern). `NoopAdapter.getConsentState()` returns `'denied'` (safest no-op default — it captures nothing regardless).
- **No** SPI verb for session id (adapter stamps `NeutralEvent.sessionId`) — Q3.
- **No** new verb for the merge (rides existing `identify()`) — Q2.

### Forward-notes folded from E2/E3 reviews

- **Injectable crypto-backed `dedupeId`** (E2-S3 reviewer forward note): the seam's `generateUuid` is a `Math.random()` v4 UUID (the seam can't reference ambient `crypto` under `lib:["ES2022"]`). The browser target HAS `crypto.randomUUID` at runtime — E4 makes `generateUuid` injectable/overridable so the browser adapter supplies a crypto-backed generator. Keep it injectable, not hard-wired; the wire mapping is E5.
- **`currentDistinctId()` is the single hook spot** (E2-S3 follow-up): the E2 improvement pass consolidated `buildEvent` + `setTraits` behind one `currentDistinctId()` reader, so E4's stub→resolver swap is one method body — no `NeutralEvent` reshape, pure population (Q1).
- **Consumer traits are gated by E3's facade allowlist before E4 registers them** (E3-S3 mirror boundary): `identify`/`setTraits`/`register*` keys are validated at the facade PRE-registration; E4 owns the register/register_once machinery and receives already-gated keys — it does not re-implement the allowlist. Library-computed keys (device/session/context) are trusted, not gated.

## Expansion path

- The shared browser substrate this epic carves (neutral event object, property-build order, persistence layer) is the base E5 (transport) and E6 (enrichment/capture) extend — additive, no seam break.
- A future non-PostHog browser adapter reuses the neutral `anonymous | identified` state, the `getDistinctId`/`reset`/`register*`/consent SPI verbs, and the persistence-mode contract unchanged; only the [WIRE] encoding (`$sesid` tuple, id-equality anonymity, cookie key names, `$anon_distinct_id` merge payload) is re-implemented behind the adapter.
- Device-id, session-id, and `dedupeId` generators are pluggable, so a consumer or future adapter can swap the id scheme without touching the identity semantics.
- **The merge RETAINS the prior anonymous id, it does not just swap (— architect 2026-07-08 refiner pass, forward-pointer).** PostHog persists the previous distinct id as the anonymous id at `identify()` (core `posthog-core.ts:374`) so a later in-flight flag call still carries `$anon_distinct_id` and keeps the merge linkage. E4 persists the prior anon id at merge time (not only the current distinct id) — the [WIRE] retention stays adapter-internal — so a future feature-flag-bootstrap slice inherits the linkage rather than rediscovering it.
